fs-finder semantics (from lib/node_modules/fs-finder/lib/Finder.js behavior, confirmed in CONTEXT.md):

finder.in(path)        → non-recursive (single directory level)
finder.from(path)      → recursive (all subdirectories)
.findFiles(mask)       → returns array of full/absolute file paths; [] if none
.findDirectories(mask) → returns array of full/absolute dir paths; [] if none
.showSystemFiles()     → include dotfiles/hidden entries (required for .git search)

mask examples in production:
  "*.zip"              glob — all zip files
  "*.ino"              glob — all Arduino sketches
  'thinx.yml'          exact name
  'environment.json'   exact name
  'environment.h'      exact name
  '.git'               exact name (dotfile/hidden dir)
  HEADER_FILE_NAME     runtime variable (exact name at call time)
  extension            runtime variable (caller supplies e.g. ".bin")

All callers use return value INLINE and SYNCHRONOUSLY — no Promises permitted.

Node engine: >=19.x (package.json engines field). The recursive: true option on
fs.readdirSync was added in Node 18.17 but is NOT available on the 19.0-19.9 line
(it was backported to 18.x before 19.x added it). Production runs Node 25 where it
works, but building against a >=19.x floor means any 19.x deployment silently
breaks. Use a VERSION-INDEPENDENT manual synchronous walk instead:
  - Maintain a stack (or queue) of directories to visit, seeded with [root].
  - Pop a dir, call fs.readdirSync(dir, { withFileTypes: true }).
  - For each Dirent: if it is a directory (and should be visited), push it onto the
    stack; if it matches the predicate, collect its absolute path.
  - Terminate when the stack is empty.
  - { withFileTypes: true } is available since Node 10.10 — safe for any >=19.x env.
fs.readdirSync(path, { withFileTypes: true }) returns Dirent objects — use for both
recursive and non-recursive cases to filter by isFile()/isDirectory().

Existing lib/thinx/util.js is web-request utilities only — do NOT add walk helpers there.

    - findFilesSync(root, mask, recursive=false, includeDotfiles=false): returns string[] of absolute paths to files matching mask under root.
    - findDirsSync(root, mask, recursive=false, includeDotfiles=false): returns string[] of absolute paths to directories matching mask under root.
    - mask is either a glob pattern containing '*' or an exact filename string.
    - Glob: only '*' wildcard supported (fs-finder used simple prefix/suffix matching, not full minimatch). Implement as: split on '*', check startsWith(prefix) && endsWith(suffix) on the basename.
    - Exact: basename === mask.
    - recursive=false: read only direct children of root using fs.readdirSync(root, { withFileTypes: true }). Do NOT descend into subdirs.
    - recursive=true: use a manual synchronous directory walk — maintain a stack seeded with [root]; pop a dir; call fs.readdirSync(dir, { withFileTypes: true }); for each Dirent, if it is a directory push it to the stack (respecting includeDotfiles); if it matches the type predicate and mask, record path.join(dir, dirent.name) as an absolute result. Continue until the stack is empty. This approach works on all Node versions >=10.10.
    - includeDotfiles=false (default): skip entries whose basename starts with '.'.
    - includeDotfiles=true: include hidden entries (required for .git directory search in repository.js).
    - If root does not exist or is not a directory, return [] (do not throw — fs-finder returned [] on missing dir; callers depend on this).
    - Export as module.exports = { findFilesSync, findDirsSync } (plain object, not a class).
  </behavior>

Export two named functions: findFilesSync and findDirsSync with the signatures and semantics described in the behavior block. ## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | Tampering | findFilesSync root parameter | accept | root is sourced from server-side app_config / build paths set by operators, not from HTTP request bodies. No new external attack surface vs. the fs-finder calls being replaced. |
| T-15-02 | Information Disclosure | Recursive walk escaping intended subtree via symlink | mitigate | The manual stack-walk uses fs.readdirSync with { withFileTypes: true } and checks entry.isDirectory() — Dirent.isDirectory() returns false for symlinks to directories (unlike isFile/isDirectory on stat). Symlinks are therefore not followed. Add a note in JSDoc that symlinks are not followed. This preserves fs-finder's existing behaviour. |
| T-15-03 | Denial of Service | Unbounded recursion on deep directory trees | accept | The manual stack-walk is iterative (not recursive), so there is no call-stack overflow risk. Traversal depth is bounded by OS filesystem limits; identical exposure to fs-finder which also walked entire trees. Scope is internal build paths not writable by API callers. |
| T-15-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan — lib/thinx/finder.js uses only Node core (fs, path). |
