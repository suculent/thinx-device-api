### 3. Procedure — Step-Numbered Operator Sequence

#### Step 1 — Sync repo to swarm host

```bash
ssh -i ~/.ssh/DOKey2 -p 2020 root@188.166.23.244
cd /mnt/gluster/deployment/swarm/  # or wherever the operator keeps the working checkout
```

#### Step 2 — Dry-run scan (mandatory; default mode)

```bash
docker run --rm --network thinx_internal \
  -v $(pwd):/work -w /work \
  -e COUCHDB_USER -e COUCHDB_PASSWORD -e COUCHDB_URL \

# expect: stdout summary with total docs scanned, matching-PII doc count, leak-shape breakdown
```

Operator reviews the report. If counts look anomalous (e.g., zero docs found, or impossibly large numbers), STOP and investigate before proceeding.

#### Step 3 — Snapshot + Apply

```bash
SNAPSHOT_PATH=/mnt/gluster/backups/managed_logs.pre-redaction-$(date -u +%Y%m%dT%H%M%SZ).jsonl

docker run --rm --network thinx_internal \
  -v $(pwd):/work -v /mnt/gluster/backups:/mnt/gluster/backups -w /work \
  -e COUCHDB_USER -e COUCHDB_PASSWORD -e COUCHDB_URL \
  node:18-alpine node scripts/redact-managed-logs.js \
    --apply \
    --snapshot-to "${SNAPSHOT_PATH}" \
    --batch-size 500
# expect: progress lines per batch; final summary with total docs redacted and snapshot path
```

Then `chmod 600 "${SNAPSHOT_PATH}"` to restrict access — the snapshot contains the raw pre-redaction PII (this is by design — it is the only rollback artifact). Document retention policy in the GDPR-posture section below.

#### Step 4 — Sample-verify

```bash
docker run --rm --network thinx_internal \
  -v $(pwd):/work -w /work \
  -e COUCHDB_USER -e COUCHDB_PASSWORD -e COUCHDB_URL \
  node:18-alpine node scripts/redact-managed-logs.js --sample 1000
# expect: exit 0; output reads "0 raw 64-char hex reset_keys + 0 raw emails found in 2000 sampled docs"
```

If exit code is non-zero, STOP. The redaction was incomplete. Investigate which doc IDs leaked (the script prints `_id`s, never the leak content). Common causes: a CouchDB write race during the apply run (re-run `--apply` to catch the stragglers — the script is idempotent), or a leak-shape the regex didn't anticipate (file an issue + add the new shape to the regex before re-applying).

Record the sample result in the GDPR-posture section below: `Date: <ISO> | N: 1000 recent + 1000 old | Result: exit 0`.

#### Step 5 — (Optional) Compact the database

```bash
docker run --rm --network thinx_internal curlimages/curl:latest \
  -X POST -H 'Content-Type: application/json' \
  "http://${COUCHDB_USER}:${COUCHDB_PASSWORD}@thinx_couchdb:5984/${THINX_PREFIX}managed_logs/_compact"
# expect: {"ok":true}
```

Compaction is OPTIONAL — CouchDB will keep both the old and new `_rev` of every redacted doc until compaction reclaims the space. Compaction can take significant time and I/O on a 658k-doc DB; schedule during a low-traffic window. Alternative: trigger via the in-process `database.js:117` call at next `thinx_api` restart (the existing `compactDatabases()` method is wired to startup or admin trigger — confirm wiring before relying on this path).

### 4. Forward-TTL Cron Recipe

The TTL field (`expire_at`, Plan 09-2) is written on every new audit record but does NOT auto-evict. Schedule a nightly cron on the swarm host scheduler to sweep expired docs.

```bash
# /etc/cron.d/thinx-managed-logs-cleanup (or via systemd timer)
# Run nightly at 03:00 UTC
0 3 * * * root docker run --rm --network thinx_internal curlimages/curl:latest \
  -s -X GET "http://${COUCHDB_USER}:${COUCHDB_PASSWORD}@thinx_couchdb:5984/${THINX_PREFIX}managed_logs/_design/cleanup/_view/expired?endkey=\"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"&include_docs=false" \
  | jq -r '.rows[] | [.id, .value] | @tsv' \
  | while IFS=$'\t' read -r id rev; do \
      curl -s -X DELETE "http://${COUCHDB_USER}:${COUCHDB_PASSWORD}@thinx_couchdb:5984/${THINX_PREFIX}managed_logs/${id}?rev=${rev}"; \
    done
```

**Note:** This recipe references a `_design/cleanup` view that emits `[expire_at, _rev]` indexed by `expire_at`. The design doc itself is OUT OF SCOPE for Phase 9 (CONTEXT.md: "CouchDB cron infrastructure depends on swarm-side scheduler + lives in `/mnt/gluster/deployment/swarm/` outside this repo"). The operator authors the design doc separately following the standard CouchDB view pattern. The recipe above is a STARTING TEMPLATE — adjust the view path and the date format to match the operator's actual design doc shape.

Alternative if the operator prefers an in-process sweep: extend `lib/thinx/database.js:compactDatabases()` in a future phase to also iterate `expire_at < now` and call `loglib.destroy(id, rev)` per match. Deferred — Phase 9 scope is the field + recipe only.

### 5. Rollback from Snapshot

If the redaction needs to be undone (e.g., a downstream auditor requests the original content for a specific incident):

```bash
# The snapshot is JSONL — one CouchDB doc per line, with original _rev preserved.
# To restore a single doc:
SNAPSHOT_PATH=/mnt/gluster/backups/managed_logs.pre-redaction-<timestamp>.jsonl
TARGET_ID="<doc _id>"

grep -F "\"_id\":\"${TARGET_ID}\"" "${SNAPSHOT_PATH}" \
  | head -1 \
  | jq '.' \
  | curl -s -X PUT \
      -H 'Content-Type: application/json' \
      -d @- \
      "http://${COUCHDB_USER}:${COUCHDB_PASSWORD}@thinx_couchdb:5984/${THINX_PREFIX}managed_logs/${TARGET_ID}?new_edits=false"
# expect: {"ok":true,"id":"...","rev":"..."}
```

For BULK rollback (full restoration of all redacted docs), POST the snapshot as a `_bulk_docs` payload in chunks of 500:

```bash
# Pseudocode — operator splits the snapshot into 500-doc chunks and POSTs each:
split -l 500 "${SNAPSHOT_PATH}" /tmp/snap-chunk-
for chunk in /tmp/snap-chunk-*; do
  jq -s '{docs: ., new_edits: false}' "${chunk}" \
    | curl -s -X POST \
        -H 'Content-Type: application/json' \
        -d @- \
        "http://${COUCHDB_USER}:${COUCHDB_PASSWORD}@thinx_couchdb:5984/${THINX_PREFIX}managed_logs/_bulk_docs"
done
```

`new_edits=false` is critical — it tells CouchDB to honor the snapshot's original `_rev` instead of creating a new one. Without this flag, the rollback creates a new revision and the redacted version stays as a previous-rev artifact (which may or may not be what the operator wants — document this nuance).

### 6. GDPR-Posture Note

Append a section titled "GDPR Posture — Historic PII Redaction" that documents:

**Scope:**
- Database: `${THINX_PREFIX}managed_logs` on production CouchDB.
- Doc count at time of remediation: ~658,808 (as of 2026-05-26 snapshot).
- Leak shapes targeted: raw 64-char hex `reset_key` substrings; raw email addresses; (best-effort) activation tokens and mailgun tokens via the same regex family.

**Method:**
- Per-doc overlay redaction via CouchDB `_bulk_docs`, preserving `_id` + `_rev` + `date` + `flags` + `owner`; replacing PII substrings in `message` (and any other string field) with literal markers `[REDACTED-RESET_KEY]` and `[REDACTED-EMAIL]`.
- Forward-going TTL: 90-day `expire_at` field added to every new audit record (Plan 09-2; configurable via `app_config.audit_retention_days`).

**Sampling Evidence Template (operator fills in after each redaction run):**

```
| Run Date (UTC) | Sample N (recent + old) | Result          | Operator |
|----------------|-------------------------|-----------------|----------|
| YYYY-MM-DD     | 1000 + 1000             | exit 0 (clean)  | <name>   |
```

**Residual Risk — explicitly accepted:**
- The redaction is DESTRUCTIVE of audit-log content. The original PII text is overwritten in CouchDB by the overlay. The ONLY artifact retaining the pre-redaction content is the JSONL snapshot file (Step 3 above), which is stored under restricted permissions (`chmod 600`) on the gluster mount.
- Snapshot retention policy (operator decision — document the choice): recommended 90 days mirroring the forward-TTL window, then secure-delete (`shred -u`). The snapshot itself is GDPR-relevant data and should not outlive its forensic purpose.
- Records older than the snapshot moment but never matched by the regex are NOT modified — they are presumed not to contain PII by the SEC-PII-01 leak-shape definition. If a new leak shape is later discovered, a follow-on phase will re-run the redaction with the expanded regex; the snapshot WILL NOT capture the original content of docs missed by the first-pass regex.

**Per-record provenance:**
- Each redacted doc carries `redacted_by: "SEC-PII-02"` and `redacted_at: <ISO timestamp>` for forensic reconstruction of which records were touched and when.
