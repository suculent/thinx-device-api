<objective>
Modify `lib/thinx/audit.js` `Audit.log()` to add a forward-going TTL field (`expire_at`) to every record it writes. The retention window is parameterized via `app_config.audit_retention_days` with a 90-day default (per CONTEXT.md decision: "parameterize with a 90-day default"). The change is strictly ADDITIVE — the function signature `log(owner, message, flag, callback)` is unchanged, the callback contract is unchanged, the existing record fields (`message`, `owner`, `date`, `flags`) are unchanged. SEC-PII-01 callers in `owner.js` (lines 169, 189, 228, 234, 476, 500, 528, 556, 614, 675, 681, etc. — the 12+1 redacted-token sites) keep working without modification because the redaction continues to happen at the CALLER, not at audit.log write-time (CONTEXT.md explicitly preserves this pattern). A new regression spec `ZZ-AuditTTLSpec.js` asserts the field shape + parameterization + default + signature-stability.

Purpose: SEC-PII-02 ROADMAP success criterion #3 mandates a TTL/retention behavior captured in a runbook. This plan ships the CODE side of that retention (the field that the runbook's cron entry will key off via a CouchDB `_design/cleanup` view emitting `[expire_at, _id]`). The TTL prevents the SEC-PII-01 leak shape from re-accumulating even if a future regression bypasses the redactors — the worst-case window becomes 90 days instead of perpetual. The 90-day window is GDPR-friendly (purpose-limitation + storage-limitation) but preserves enough horizon for security-incident forensics.

Output: One file modified (`lib/thinx/audit.js`), one new spec (`spec/jasmine/ZZ-AuditTTLSpec.js`). One atomic GPG-signed commit: `feat(SEC-PII-02): add expire_at TTL field to audit log writes (90-day default)`.
</objective>

Extend `Audit.log(owner, message, flag, callback)` so that the `record` object now also carries an `expire_at` field. Strict invariants:

1. The function signature `log(owner, message, flag, callback)` is UNCHANGED. No new positional args. No new named opts.

Implementation:

a. Top of `Audit.log`, after computing `mtime`, look up the retention horizon:

```
let retentionDays = 90;
try {
  const cfg = (typeof Globals.app_config === "function") ? Globals.app_config() : null;
  if (cfg && typeof cfg.audit_retention_days === "number" && cfg.audit_retention_days > 0) {
    retentionDays = cfg.audit_retention_days;
  }
} catch (_e) {
  // app_config unavailable (e.g., very early boot, certain test contexts) — silently fall back to 90 days.
  retentionDays = 90;
}
```

Wrap in try/catch because `Globals.app_config()` MAY throw in some early-boot or test contexts (it calls `_public.load()` which can fail). A throw here MUST NOT break audit writes — fall back to 90.

b. Compute expiry:

```
const expire_at = new Date(mtime.getTime() + retentionDays * 24 * 60 * 60 * 1000);
```

Use `mtime.getTime()` so `expire_at` is deterministically `retentionDays` after the record's own `date`, NOT after the read of `Date.now()` (which could differ by a microsecond and complicate test assertions).

c. Add `expire_at` to the record AFTER `flags`:

```
let record = {
  "message": message,
  "owner": owner,
  "date": mtime,
  "flags": Array.isArray(flag) ? flag : [flag],
  "expire_at": expire_at
};
```

d. NO changes to `loglib.insert(record, mtime, ...)` call. NO changes to the callback wrap.

e. Do NOT touch `Audit.fetch()` — out of scope for this plan.

Module-level imports: `Globals` is ALREADY imported on line 5 (`const Globals = require("./globals.js");`) and the prefix is used on line 6. No new requires needed.

Caller invariant — DO NOT MODIFY ANY CALLER. The 12+1 `alog.log(...)` sites in `owner.js` (and any other site) must remain bit-identical. This is an audit.js-internal change only.
