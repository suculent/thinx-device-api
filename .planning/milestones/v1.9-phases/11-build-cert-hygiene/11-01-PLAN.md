<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → script | Operator invokes script with CLI args; args become Bash variables interpolated into `docker buildx build -t "$OWNER/base:$TAG"`, `git commit -m`, and `docker push`. |
| script → docker daemon | Script issues `docker buildx build` + `docker push`; mismatched/typo'd `--owner` could push to the wrong namespace. |
| script → git index | Script issues `git commit -S` (GPG-signed); a bug in the staging step could commit unintended files. |
| script → npm registry | `npm install . --omit=dev` + `npm audit fix` pull from npm; outside the script's control but inherits npm's signature/integrity checks (package-lock.json regen on each run). |
| script → package.json | `npm version patch --no-git-tag-version` mutates `package.json` version field at the repo root. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | Tampering | `--owner` arg | mitigate | Bash double-quote interpolation `"$OWNER/base:$TAG"` prevents word-splitting + glob expansion. Operator-only invocation context (local dev machine, not CI); typo risk accepted (would push to a non-owned Docker Hub namespace → push fails with 401, surfacing the mistake). |
| T-11-02 | Repudiation | Atomic commit | mitigate | `git commit -S` (GPG-signed per project convention). The commit subject is fixed to `chore: base version bump` so its provenance is auditable in `git log`. |
| T-11-03 | Information Disclosure | Pre/post digest echo | accept | Digests are public artifact identifiers (no secret material). Logging to stdout is intentional for operator audit trail. |
| T-11-04 | Denial of Service | Unconditional `rm -rf ./node_modules/` + `rm -rf ./package-lock.json` | accept | Destructive but scoped to `base/` subdirectory (relative paths `./node_modules/` and `./package-lock.json`). The original script already does this — preserving the behavior. The Bash `set -e` ensures that if `rm -rf` itself fails (e.g., read-only fs), the script aborts before the docker build runs. |
| T-11-05 | Elevation of Privilege | `npm install` running arbitrary postinstall hooks from transitive deps | accept | Same risk as the existing script + the repo's normal `npm install` lifecycle. `npm audit fix` is preserved. No new attack surface introduced by the rewrite. |
| T-11-06 | Tampering | `node -p "require('./package.json').version"` reading version | mitigate | `node -p` interprets its argument as JavaScript; embedding `package.json` content via `require()` is the safe form (no shell-string-parsing of JSON). |
| T-11-SC | Tampering | npm install in the script | mitigate (deferred) | Package Legitimacy Audit not in RESEARCH.md for this phase (no new packages introduced; the script only consumes `package.json` deps that already passed prior phases' gates). Inherited posture from the repo's existing npm supply-chain stance (lockfile committed, audit run). |

**Disposition notes:**
- `T-11-04` (unconditional rm -rf) is preserved-from-original behavior; flagging here for operator awareness rather than changing scope.
- `T-11-05` (npm install lifecycle) is inherited project posture; rewrite does not widen this surface.
- `T-11-SC` (supply chain): no new packages introduced by this plan — the rewrite is pure shell logic over existing tooling. No legitimacy checkpoint needed.
</threat_model>
