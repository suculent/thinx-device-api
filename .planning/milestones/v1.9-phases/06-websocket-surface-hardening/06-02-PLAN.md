Express + connect-redis Set-Cookie attribute contract:
- When `cookie.httpOnly: true`, the response `Set-Cookie` header contains the `HttpOnly` attribute (case-sensitive in the header — Express writes it as `HttpOnly`, RFC 6265).
- When `cookie.httpOnly: false`, the response `Set-Cookie` header OMITS the `HttpOnly` attribute (it's not written as `HttpOnly=false`; the absence of the flag is how `false` is encoded).
- The regression spec MUST therefore assert that the `HttpOnly` substring appears in the `Set-Cookie` header for `x-thx-core`. Use chai-http v4's `res.headers['set-cookie']` (which is an array of strings).

|----------|-------------|
| Browser JS → `document.cookie` | When `httpOnly: false`, malicious JS (XSS payload) can read the session cookie. When `httpOnly: true`, the browser refuses to expose the cookie to JS — narrows XSS-stealable-cookie surface. |
| Client → server (`Cookie:` header) | Both `httpOnly: true` and `httpOnly: false` result in the cookie being sent in the request header. So server-side cookie reads in `request.headers.cookie` are unaffected. The WS upgrade at thinx-core.js:471 + :621 uses server-side reads. |
| Express session config → Set-Cookie response header | The `cookie.httpOnly` config value controls whether the `HttpOnly` attribute is written into the `Set-Cookie` response header. Regression spec asserts on this exact header. |
