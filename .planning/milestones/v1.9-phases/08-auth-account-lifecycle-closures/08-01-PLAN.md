<objective>
Implement AUTH-REACTIVATE-01 — an admin-only `POST /api/v2/admin/user/:id/reactivate` endpoint that flips a soft-deleted user's `deleted` flag from `true` to `false`, restoring their login ability. The new route must reuse the existing `requireAdmin` middleware (no new auth mechanism) and the existing `userlib.atomic("users", "edit", ...)` write pattern (mirror of `Owner.delete` at owner.js:693-700). A regression spec covers (a) unauthenticated → 401, (b) authenticated-non-admin → 403, (c) admin → success and `deleted` flag flipped, (d) the soft-delete login lockout at router.auth.js:189-191 still rejects genuinely-deleted accounts.

Purpose: Close the v1.0 UAT gap where soft-deleted users currently require direct CouchDB mutation to be reactivated — there is no API surface today. This phase ships the admin-only path (operator decision per D-01 in CONTEXT.md); the self-serve email-link flow stays explicitly deferred.

Output: One new admin route + one new regression spec. Two atomic GPG-signed commits (handler, then spec — or one combined commit if execute-plan prefers handler+spec together; the executor decides per existing project convention).
</objective>

From lib/middleware/requireAdmin.js:

```
module.exports = function (app) {
  return function requireAdmin(req, res, next) {
    if (!Util.validateSession(req)) return res.status(401).end();
    const owner = sanitka.owner(req.session.owner);
    if (typeof (owner) === "undefined") return res.status(401).end();
    app.owner.profile(owner, (success, profile) => {
      if (!success || !profile || profile.admin !== true) {
        return res.status(403).end();
      }
      next();
    });
  };
};
```

Behavioral contract:
- requireAdmin returns 401 if no valid session OR sanitized owner is undefined.
- requireAdmin returns 403 if profile.admin !== true.
- Only `next()` reaches the route handler.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Untrusted HTTP client → POST /api/v2/admin/user/:id/reactivate | Any internet client may attempt this route; auth and authorization must happen at the route layer. |
| Admin session → userlib.atomic write | Admin identity comes from session/JWT — confidence is only as strong as the session-handling pipeline (validated by `Util.validateSession` + `requireAdmin`). |
| URL path parameter `:id` → CouchDB document key | The `:id` value flows from the URL into a CouchDB document update via `userlib.atomic`. Must be sanitized. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08.1-01 | Elevation of Privilege | POST /api/v2/admin/user/:id/reactivate | mitigate | Route registered with `requireAdmin` middleware (lib/middleware/requireAdmin.js) — non-admin returns 403, unauthenticated returns 401. Locked by spec scenarios 1 and 2. |
| T-08.1-02 | Tampering | `:id` path parameter | mitigate | `sanitka.owner(req.params.id)` rejects malformed IDs before any CouchDB call; undefined → 400 `missing_owner`. Same pattern as impersonate handler (router.admin.js:34, 49). |
| T-08.1-03 | Repudiation | Admin reactivation action | mitigate | `alog.log(admin_owner, "Reactivation succeeded/failed for " + target_owner, ["admin", "reactivate", ...])` — audit log records both admin actor and target on every reactivation attempt. |
| T-08.1-04 | Denial of Service | Mass reactivation attempts | accept | The route is admin-only — DoS surface is limited to the existing admin-account population. No rate limiting added in this phase. |
| T-08.1-05 | Information Disclosure | Error message leakage | mitigate | All failure paths use `Util.failureResponse(res, code, "string_literal")` with whitelisted response strings (`missing_owner`, `user_update_failed`). The underlying `err` object from `userlib.atomic` is NOT serialized into the response body (matches the SEC-PII-01 redaction discipline established in Phase 2 / followed in Phase 7). |
| T-08.1-06 | Spoofing | Session hijack → admin impersonation | accept | Out of scope; covered by the existing session/JWT pipeline (Phase 2 SEC-PII-01 + Phase 6 WS auth). This phase does NOT change session-handling semantics. |
| T-08.1-07 | Tampering | Soft-delete login gate at router.auth.js:189-191 | mitigate | NO modification to router.auth.js in this phase (must_have #4). Spec scenario 6 asserts the gate still returns 403 `user_account_deactivated` for `deleted: true` users — locking the gate's intact behavior. |
| T-08-SC | Tampering | npm/pip/cargo installs | accept | No new package installs in this phase (handler uses already-required modules in router.admin.js: Util, Sanitka, AuditLog, requireAdmin). RESEARCH.md Package Legitimacy Audit therefore N/A. |
