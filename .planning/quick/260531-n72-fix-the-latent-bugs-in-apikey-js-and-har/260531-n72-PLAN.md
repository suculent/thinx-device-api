<objective>
Fix the three latent bugs that converged to produce the 2026-05-31 thinx_api OOM
(13+ orphan MQTT users, exit 137 after 5 minutes of looping). The three concerns
are atomic and independent, so they ship as three tasks inside a single plan:

1. apikey.js no longer conflates "Redis unreachable" with "key not found", no
   longer double-calls its callback on Redis set error, and rate-limits the
   create-default-key retry loop.
2. The node-redis v5 client (configured in lib/thinx/globals.js, instantiated
   in thinx-core.js) reconnects forever with bounded exponential backoff and
   degrades gracefully (no crash) on 'error'/'end'.
3. A new lib/thinx/redis-health.js posts a debounced disconnect message and a
   single recovery message to Slack using the EXISTING slack-notify integration
   already wired in lib/thinx/notifier.js (env var SLACK_WEBHOOK — no new var).

Purpose: prevent the OOM regression; close the orphan-MQTT-user leak; give the
operator a Slack signal the moment Redis goes dark instead of waiting for a
deploy-loop alert.

Output: patched apikey.js + owner.js + globals.js + thinx-core.js; new
lib/thinx/redis-health.js; extended ApikeySpec.js; new RedisHealthSpec.js.
</objective>

APIKey contract (lib/thinx/apikey.js):
- `constructor(redis)` — redis is the legacy callback-style client.
- `save_apikeys(owner_id, api_key_array, callback)` — callback(success: bool, array)
  BUG: today the error branch falls through and calls callback twice.
- `create(owner_id, alias, callback)` — callback(success: bool, array_or_error)
  BUG: today `redis.get` error branch jumps to `save_apikeys([new_key])` (treats
  Redis error as "key missing"). MUST distinguish ClientClosedError /
  connection errors and propagate `callback(false, "redis_unavailable")`.

Auth contract (lib/thinx/auth.js):
- `add_mqtt_credentials(_username, password, callback)` — callback() (no args)
- `revoke_mqtt_credentials(username)` — synchronous-ish del, used for compensation

Owner contract (lib/thinx/owner.js):
- `create_mqtt_access(owner_id, DEFAULT_APIKEY_NAME, callback)` — callback(success, key_obj)
  Today calls apikey.create then auth.add_mqtt_credentials in the success
  branch. The ORDER is already save-then-mqtt — the leak comes from apikey.create
  reporting fake success on Redis error. Task 1's apikey.create fix is the
  primary cure. Task 1 also adds an explicit compensation hook here: if apikey.create
  returns "redis_unavailable", skip MQTT and bubble the error; if MQTT add itself
  fails (the bcrypt path swallows errors today), call auth.revoke_mqtt_credentials.

Slack integration (REUSE — do not roll a new HTTP call):
- `lib/thinx/notifier.js` already uses `SlackNotify(process.env.SLACK_WEBHOOK)`
  from the `slack-notify` npm package (package.json: "slack-notify": "^2.0.6").
- `lib/thinx/notifier.js:42-54` (notifyAppStart) is the exact pattern to mirror:
  read SLACK_WEBHOOK, log+return if missing, otherwise `SlackNotify(webhook).send({text, username, icon_emoji, channel: "#thinx"})`.
- DO NOT add a new env var. The existing var is `SLACK_WEBHOOK` (no `_URL` suffix).

node-redis v5 events (per createClient instance):
- `'error'` (err) — fires on socket errors, command failures, etc. Already
  handled at thinx-core.js:102 with a console.log. Wrap it via RedisHealth.attach.
- `'end'` (no args) — fires when the connection ends and reconnect is not
  pending (rare with infinite reconnect, but possible during shutdown).
- `'ready'` (no args) — fires after successful connect/reconnect. Use as the
  recovery signal.
- `'reconnecting'` — fires before each retry attempt. Useful for debug log
  only; do NOT post Slack on this (would defeat debouncing).
