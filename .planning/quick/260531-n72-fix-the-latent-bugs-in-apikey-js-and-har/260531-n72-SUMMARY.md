# Quick Task 260531-n72: Redis Hardening + Slack Outage Notifier — Summary

Close the three latent bugs that converged to produce the 2026-05-31
`thinx_api` OOM (13+ orphan Mosquitto users, exit 137 after a 5-minute
loop). The fixes ship as three atomic commits in a single plan.

## Tasks Completed

### Task 1 — `apikey.js` Redis-error disambiguation + circuit breaker + `owner.create_mqtt_access` compensation

**Commit:** `78e2fa1e`
**Files:** `lib/thinx/apikey.js`, `lib/thinx/owner.js`, `spec/jasmine/ApikeySpec.js`

- New top-level `isRedisUnavailable(err)` helper in `apikey.js` detects
  `ClientClosedError`, `SocketClosedUnexpectedlyError`,
  `DisconnectsClientError`, and the common connection-level error codes
  (`ECONNREFUSED` / `ECONNRESET` / `ETIMEDOUT`), plus an `err.message`
  regex fallback. Bias toward MORE matches — a false positive triggers a
  retry; a false negative recreates the OOM-causing fall-through.
- `APIKey.create()`: on `isRedisUnavailable(error)` from `this.redis.get`,
  surfaces `callback(false, "redis_unavailable")` instead of falling
  through to `save_apikeys`. The stale comment "keys do not exist, save
  new array" is removed (was load-bearing documentation of the bug).
- Static `APIKey._lastDefaultKeyAttempt = new Map()` declared at module
  bottom. The breaker fires at the TOP of `create()` for alias
  `"Default MQTT API Key"`: if the previous attempt fired within 60s, it
  short-circuits with `callback(false, "redis_unavailable_backoff")` and
  never touches `this.redis`. Other aliases (user-driven create calls)
  are unaffected.
- `APIKey.save_apikeys()`: added missing `return` to the error branch so
  the callback fires exactly once on `redis.set` error. Previously it
  fell through and called `callback(true, ...)` after the error path —
  classic double-callback bug.
- `Owner.create_mqtt_access()`:
  - On `(false, "redis_unavailable")` / `(false, "redis_unavailable_backoff")`,
    propagates `callback(false, reason)` WITHOUT calling
    `auth.add_mqtt_credentials` — closes the leak at a second
    independent layer.
  - On success, wraps `auth.add_mqtt_credentials` in a once-fired settle
    guard with a 5s watchdog. If `add_mqtt_credentials` throws
    synchronously or its callback never fires, the (possibly
    half-written) Mosquitto credential is revoked via
    `auth.revoke_mqtt_credentials`. No orphan is left behind.
- `spec/jasmine/ApikeySpec.js`: appended three stub-based `it()` blocks
  (R1, R2, R3) using a `makeClosedStub()` helper. They run without a live
  Redis and assert the Redis-closed path, the breaker, and the
  single-callback invariant respectively.

### Task 2 — `globals.js` infinite-reconnect Redis strategy

**Commit:** `ca984ed2`
**Files:** `lib/thinx/globals.js`

- `redis_reconnect_strategy` rewritten:
  - **Test envs** (`CIRCLE_USERNAME` set OR `ENVIRONMENT === "test"`)
    keep the old short-cap behavior — Error after 5 retries, Error on
    `ECONNREFUSED` / `ECONNRESET` / `ETIMEDOUT` — so the CI test suite
    remains bounded.
  - **Production / staging** never return an Error while the process is
    alive. Returns `Math.min(30_000, 100 * Math.pow(2, retries))` —
    100 ms, 200 ms, 400 ms, 800 ms, ..., capped at 30 s forever.
- Each retry beyond the first logs a single info breadcrumb so the
  deploy log shows backoff is engaged.
- Inline comment explains the OOM context so future readers don't
  "helpfully" reintroduce the terminal-Error branches.

### Task 3 — `lib/thinx/redis-health.js` Slack-debounced outage notifier + `thinx-core.js` wire-up

**Commit:** `dd554716`
**Files:** `lib/thinx/redis-health.js` (new), `thinx-core.js`, `spec/jasmine/RedisHealthSpec.js` (new)

- New single-file module exports `attach(client, opts?)` returning
  `{ detach() }`. Subscribes to `error`, `end`, and `ready` events on
  the node-redis v5 base client.
- **First disconnect** within the (default 5-minute) `debounceMs` window
  posts `"[THiNX API] Redis client disconnected: <reason>"` via
  SlackNotify. **Subsequent disconnects** within the window are
  suppressed. **First `ready` event** after an outage posts
  `"[THiNX API] Redis client reconnected, healthy"` exactly once;
  `lastDownPostedAt` is then reset to 0 so a future outage posts
  immediately.
- Reads the webhook from `opts.webhook ?? process.env.SLACK_WEBHOOK`
  (SAME env var as `lib/thinx/notifier.js` — no new env var introduced).
  Missing webhook causes a single startup warning and thereafter no-ops.
- Uses the existing `slack-notify` npm package (already in
  `package.json`). Mirrors the lazy-require pattern from
  `lib/thinx/notifier.js`. SlackNotify.send failures are caught and
  logged so the notifier itself can never crash the process.
- `thinx-core.js`: added `const RedisHealth = require('./lib/thinx/redis-health');`
  next to the other lib requires (top of file) and called
  `RedisHealth.attach(app.redis_store_client)` immediately after the
  existing `on('error', ...)` registration AND before `.connect()`, so
  the very first connection failure is captured. The legacy
  `console.log` error handler is intentionally kept (comment in the
  diff documents the choice).
- `spec/jasmine/RedisHealthSpec.js`: four hermetic cases (H1, H2, H3,
  H4) using an `EventEmitter` stub and an injected SlackNotify factory
  — no live Redis, no live network.

## Deviations from Plan

**None of substance.** Two minor implementation notes worth recording:

1. The static map `APIKey._lastDefaultKeyAttempt` is exposed via
   `module.exports._lastDefaultKeyAttempt = new Map()` declared OUTSIDE
   the class body rather than as a class static field. This was a
   stylistic choice for consistency with the existing CommonJS pattern
   in the codebase (no `static` keyword used elsewhere in
   `lib/thinx/`). The spec uses `APIKey._lastDefaultKeyAttempt.delete(...)`
   exactly as the plan asked, and the breaker logic in `create()`
   reads/writes through `APIKey._lastDefaultKeyAttempt` so both internal
   and external access paths agree.
2. `RedisHealth.attach` keeps the existing
   `app.redis_store_client.on('error', console.log)` line in
   `thinx-core.js` alongside the new attach() — the plan permitted
   either approach ("Keep the existing console.log error handler OR
   fold it into the new module — implementer's choice"). The console
   trace remains useful for tail-following the deploy log; RedisHealth
   handles the Slack throttling.

## Operator Notes

- `SLACK_WEBHOOK` is the same env var already used by
  `lib/thinx/notifier.js`. If it was previously set for app-start
  notifications, Redis outage notifications are automatically active
  with no further configuration. If it is unset, both notifiers log a
  single startup warning and no-op (never throw).
- The circuit breaker is **per-process**; restarting the API resets
  the per-owner `lastDefaultKeyAttempt` map. This is intentional — a
  fresh process boot is a legitimate "try again" signal.
- The 30-second backoff cap means worst-case, the API reconnects to
  Redis within 30s of any transient outage without operator
  intervention. The single Slack message at outage start + single
  recovery message at recovery preserves alert-fatigue hygiene.
