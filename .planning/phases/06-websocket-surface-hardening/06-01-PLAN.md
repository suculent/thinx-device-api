<objective>
Close the WebSocket resource-cleanup gap (REFACTOR-03) by adding a raw-socket `close` handler inside the upgrade flow at `thinx-core.js:459-501`. Today, the raw socket is inserted into `socketMap` at line 485 BEFORE `wss.handleUpgrade()` runs (lines 487-497). If the upgrade aborts mid-flight — client disconnect, network error, or the existing `_upgradeException` catch — the raw socket's `socketKey` stays in `socketMap` permanently because the wss-level `ws.on('close')` at line 597-599 is only attached AFTER the upgrade succeeds (inside `initSocket`). The same-`socketKey` duplicate-upgrade guard at line 463 stays "armed" with no live connection, blocking legitimate reconnects and growing `socketMap` monotonically.

The fix is the single line locked in CONTEXT.md: register a `socket.on('close', () => socketMap.delete(socketKey))` immediately after the `socketMap.set` call. The wss-level handler at line 597-599 stays — it covers the post-upgrade lifecycle and does NOT double-delete (Map.delete is idempotent).

Purpose: Eliminate the raw-socket map leak. Match REFACTOR-03's literal requirement language ("per-connection resources are released deterministically").

Output: A ~3-line code addition in `thinx-core.js`, a new Jasmine spec `spec/jasmine/ZZ-WebSocketLifecycleSpec.js` asserting `socketMap` returns to 0 after an aborted raw-socket upgrade, and one atomic GPG-signed commit: `refactor(REFACTOR-03): raw-socket close handler in WS upgrade flow`.
</objective>

- `socket.on('close', listener)` — emitted once when the socket is fully closed. The `had_error` boolean argument is irrelevant for this use case.
- Map.delete(key) is idempotent — calling it on an already-removed key is a no-op (returns `false`). So the wss-level `ws.on('close')` running after the raw-socket close is safe.
- The `socketKey` variable is captured by closure — both handlers operate on the same key string.
