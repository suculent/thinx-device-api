<objective>
Land the two atomic backend fixes for G8 / AUTH-API-01:

1. **Bearer-null guard** in `lib/router.js` — stop treating literal `Authorization: Bearer null` (and `Bearer undefined` / `Bearer `) as a real JWT. Treat as no-token; fall through to the existing cookie / no-auth path. This is the root cause of the rtm-observed 403 on `POST /api/v2/password/reset` from the Vue console (Vue `core/api.js:53-57` unconditionally sets `Authorization: 'Bearer ' + this.refreshToken`, which serializes to `"Bearer null"` when logged out).

2. **No-enumeration normalization** in `lib/router.user.js` — `postPasswordReset` currently returns 200 for registered emails and 401 + `{success:false,response:"email_not_found"}` for unregistered emails, which violates AUTH-API-01 criterion (b) "no enumeration." Normalize to always respond 200 with the standard envelope. Update the existing assertion at `spec/jasmine/ZZ-AppSessionUserSpec.js:170-180` (which asserts the old 401 + `email_not_found` body) to match.

Purpose: Close the user-visible 403 on rtm that blocks every "Forgot password?" flow and close the V7 enumeration leak that AUTH-API-01 explicitly forbids. The fix is symmetric for the v2 endpoint and the legacy `/api/user/password/reset` alias because both routes mount the same `postPasswordReset` handler (D-01 — legacy alias parity).

Output:
- `lib/router.js` with the inserted token-validity guard (one atomic commit).
