# Phase 18 — Plan 18-01: Complete GDPR Purge Orchestrator

**Requirement:** SEC-PII-03 · **Issue:** #353 · **Risk:** HIGH (irreversible deletes)

## Approach
One orchestrator (`lib/thinx/owner_purge.js`, class `OwnerPurge`) that purges every owner-scoped artifact, wired into both the on-demand `DELETE /api/v2/gdpr` and the scheduled expiry path. Path-safety is the gate: a pure, unit-tested function blocks any filesystem delete that could escape the owner subtree.

## Tasks (TDD where pure)

1. **Path-safety core (TDD first).** `OwnerPurge.safeOwnerPath(data_root, base_segment, owner)`:
   - sanitize `owner` via `Sanitka.owner` (`^[a-z0-9]{64,}$`); invalid → `null`
   - build `path.join(data_root, base_segment, owner)`; require `resolve(path)` to start with `resolve(data_root)+sep` AND `basename === owner`; else `null`
   - Unit spec `spec/jasmine/OwnerPurgeSpec.js` (no services): valid → path; `..`, `/`, empty, <64-char, uppercase → null. **Red before impl.**

2. **`rsakey.js` → `revokeAllForOwner(owner)`** — delete priv+pub for every `getKeyPathsForOwner(owner)` entry; return deleted list. (Existing `revokeUserKeys` indexes by filename list; this deletes the full set.)

3. **`buildlog.js` → `purgeOwner(owner, cb)`** — `latest_builds` view, filter `doc.owner === owner`, `buildlib.destroy` each (reuse `prune` shape), regardless of age.

4. **`OwnerPurge.purge(owner, callback)`** — order: revoke devices+MQTT (via injected `devices` using `devices.list`→`revoke`) → purge builds → `revokeAllForOwner` → remove deploy tree (`data_root`+`deploy_root`) + repo tree (`data_root`+`build_root`) via `safeOwnerPath` + `fs-extra.remove` → Redis `del("ak:"+owner)` and `del` each `/<owner>/*` key → destroy user doc LAST. Each step try/catch, collect per-step report, `alog` audit line. Idempotent (ignore ENOENT/missing).

5. **Wire `router.gdpr.js` `revokeGDPR`** — replace inline device-loop + `expire(1)` + early user destroy with `OwnerPurge.purge(owner, …)`; respond from the report.

6. **Wire `gdpr.js`** — on confirmed-expired owner, invoke `OwnerPurge.purge` (full purge, not just `delete_expired` mark). Construct `devices` from `app.messenger`/`app.redis_client`.

7. **Integration spec** — un-skip the `xit` `DELETE /api/v2/user` / extend `ZZ-AppSessionUserV2DeleteSpec.js`: seed owner with devices/keys/Redis/fs, purge, assert each store empty + `KEYS /<owner>/*`/`ak:<owner>` empty. (Runs in CircleCI — needs Redis/CouchDB.)

## Success criteria
1. fs delete cannot escape owner subtree (pure unit test proves `..`/`/`/empty/short → null).
2. RSA key files, deploy+repo trees, builds, devices, Redis keys all removed; user doc destroyed last.
3. Redis uses `del` not `expire(1)`.
4. Orchestrator reused by scheduled path (single code path).
5. Idempotent + per-step audit; integration spec green in CI.
